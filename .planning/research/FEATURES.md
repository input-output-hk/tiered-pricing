# Feature Landscape — CIP Evidence Package (menu-style mechanism proposal)

**Domain:** CIP-grade evidence base supporting a multi-option ("menu") transaction-fee mechanism proposal responding to [CPS-0023](../../docs/phase-2/CPS-0023/README.md).
**Researched:** 2026-05-15
**Milestone shape:** SUBSEQUENT — simulator + audit trail exist; this milestone produces *evidence artefacts* the user will cite when authoring the CIP.

## Reading guide

This document is a feature catalogue for the *evidence package*, not for the simulator. "Build cost" estimates assume the existing audit-trail artefacts (`docs/phase-2/validity-threats.md`, `cardano-realism-audit.md`, `.planning/family-b-decision-2026-05-14.md`, `.planning/mechanism-welfare-impact-2026-05-14.md`, `REVIEW.md` Fix Status, 7 numbered spikes) are the starting point. Every "table stakes" item below corresponds to something the CIP author or a CIP reviewer will look for once and either find or not find.

Categories are mutually exclusive: a feature is a table stake (reviewer expects it; absence is a defect), a differentiator (reviewer is pleasantly surprised; presence increases trust beyond the floor), or an anti-feature (looks like rigour but isn't, and crowds out the real thing).

## Table Stakes

Features whose absence will draw reviewer pushback or — under CIP-1's Rationale-and-evidence requirements — block "Proposed → Active" progression. Each has a precedent in CIP-1, an existing high-quality CIP, or peer-reviewed mechanism-design publications.

| # | Feature | Why expected (precedent) | Complexity | Current state |
|---|---------|--------------------------|------------|--------------|
| 1 | **Claim → backing-job mapping table** | CIP-1 Rationale requires "evidence of consensus … significant objections or concerns"; menu CIPs must let a reviewer audit each option's promised behaviour. Roughgarden's EIP-1559 paper explicitly maps each game-theoretic claim (DSIC, MMIC, OCA-proofness) to the mechanism behaviour that licenses it. | Medium (~½ day) | **Active requirement** — listed in `PROJECT.md` as a deliverable; not yet built |
| 2 | **Variance bands per claim, not point estimates** | Single-seed numbers are the canonical "looks rigorous but isn't" failure. Williams 2024 (transparent reporting for simulation studies) lists "report uncertainty across replications" as a foundation. The current evidence base reports single-seed (seed=1) welfare aggregates in `family-b-results-table` and `mechanism-welfare-impact`. | Medium (compute-bound; recipe exists) — re-run canonical menu-item jobs at N seeds, report median + IQR rather than 95% CI (3-seed median + IQR is what `validity-threats.md` already prescribes) | **Active requirement** in `PROJECT.md`; not yet executed |
| 3 | **Sensitivity sweeps per controller knob (calibration-knob audit)** | CIP-1 Rationale demands "alternate designs considered." For a controller, each knob is implicitly an alternate-design dimension. EIP-1559 deployment was preceded by Roughgarden's sweep over the base-fee step size; Cardano CIP-9 records the initial-parameter-setting rationale explicitly. | Low for sweeps already pinned in goldens-pinned suites (the `phase-2-eip1559-robustness` / `-smoothing` suites already cover D × target and window-length); Medium for the 4 unanchored knobs flagged in `cardano-realism-audit.md` | **Partially built** — `phase-2-eip1559-robustness` (D×target sweep) and `phase-2-eip1559-smoothing` (window-length sweep) are in tree; 4 controller knobs remain unanchored per audit (window=32; multiplier_floor ∈ {4, 16}; lane-signal source). |
| 4 | **Calibration provenance per parameter value** | "Where did this number come from?" is reviewer-question #1 for any numerical knob. CIP-9 (Shelley protocol parameters) is the precedent: every initial value cites the Incentivised Testnet, Haskell Testnet, SPO benchmarking, or security concern as its source. CIP-0164's Table 7 follows the same pattern. | Low (~½ day) — `cardano-realism-audit.md` already does this for most knobs; the rewrite (`PROJECT.md` Active) finishes the job by removing the 2026-05-13 annotation banner and folding it into authoritative voice | **Mostly built** — `cardano-realism-audit.md` exists, but reads as annotated rather than authoritative per `PROJECT.md` Active item |
| 5 | **Explicit "known limitations / disclosure" section** | Both CIP-1 Rationale ("backward compatibility … impacts") and the Roughgarden tradition (named impossibility theorems frame what the mechanism *cannot* deliver). For a menu CIP, the disclosure section is what lets each option's not-claimed properties stay un-claimed without reviewer suspicion. | Low (~1 day) — `validity-threats.md`'s "Resolved 2026-05-14" idiom is the template; aggregate over the 5 menu items into a single disclosure list with severity and mitigation | **Partial** — disclosure content exists scattered across `cardano-realism-audit.md`, `validity-threats.md`, `REVIEW.md`. Not yet consolidated into one CIP-citeable section |
| 6 | **Reproducibility instructions for every cited number** | CIP-1 Acceptance Criteria require "observable metrics or deliverables." Williams 2024 reproducibility checklist requires data + code + environment + seeds + commit hash. The existing suite-level golden hashes + `experiment-suite verify` subcommand are *already* the strongest part of this story — the gap is exposing them to readers as a single recipe. | Low (~½ day) — recipe is `cd sim-rs && cargo run --release --bin experiment-suite -- run <suite>.yaml` + `verify`; the suite-level golden hashes (`parameters/phase-2-sweep/suites/.goldens/`) lock the numbers; commit hash + `topology-realistic-100.yaml` lock the inputs | **Mostly built** — recipe exists in `CLAUDE.md` "Running the suites" section; needs a one-page reader-facing distillation that maps "every welfare number in the CIP" to "run this command + check this hash" |
| 7 | **Realism-risks register (one-row-per-risk)** | The user's stated worry — "code changes are large, hard to feel confident what we have is representative" — is the reviewer's worry too. The audit pattern `validity-threats.md` already uses ("category | impact | evidence | verdict") is well-known from threats-to-validity epidemiology and Sjøberg's 2023 construct-validity reporting work. | Medium (~1–2 days) — pattern exists in `validity-threats.md` but is suite-indexed; needs re-indexing by risk-category with explicit verdict (LIVE/DORMANT/MITIGATED/DISCLOSED) | **Active requirement** in `PROJECT.md` — listed as deliverable, not yet built |
| 8 | **At least one comparator baseline per menu option** | CIP-1's "alternate designs considered" requires *comparison*, not just enumeration. The Roughgarden paper treats EIP-1559 vs first-price auction vs tipless mechanism as the baseline triplet. The phase-2 evidence already retains EIP-1559 as a *control* (per `PROJECT.md` Out-of-Scope #5: "Cutting the EIP-1559 suites — retained as control evidence"). Each two-lane menu item needs its single-lane EIP-1559 contrast number cited. | Low — already there; what's missing is the explicit table contrasting each two-lane variant's headline welfare number against the single-lane baseline at matched demand profile | **Mostly built** — the 7 goldens-pinned suites include both single-lane and two-lane jobs; needs the contrast table assembled |
| 9 | **Statement of what each menu option *trades*** | Menu CIPs differ from single-recommendation CIPs precisely in that the reviewer expects the trade-off to be made explicit. CPS-0023 itself models this — its "Goals," "Constraints," and "Non-Goals" sections force the proposed solution to be specific about *what is and isn't* delivered. Each menu option's "this gives you X at cost Y" needs a one-line crystallisation. | Low (~½ day) — the per-suite Q-A framing in `validity-threats.md` is the raw material; just needs lifting and tightening | **Partial** — material exists, not yet in menu-option-comparison form |
| 10 | **Pinned commit / artefact reference** | Reproducibility precedent: every reproducible-paper checklist (Williams 2024, the "five pillars" framework) requires a frozen artefact citation. `experiment-suite verify` + the suite goldens already provide the bit-stable check; the CIP needs to cite a specific commit hash (and the topology snapshot's epoch). | Low (~10 min) — git tag the publication commit; cite the `topology-realistic-100.yaml`'s epoch-582 snapshot date | **Easy to add** — not done because no CIP citation exists yet |

## Differentiators

Features not strictly required for CIP-1 sufficiency, but which materially raise the evidence package above competent-and-disclosed into "this is hard to argue against." For a menu CIP responding to an open CPS, each of these moves the evidence base from "defensible" to "definitive on its claimed scope."

| # | Feature | Why it raises trust (precedent) | Complexity | Recommendation |
|---|---------|--------------------------------|------------|----------------|
| D1 | **Counterfactual welfare numbers ("what if we did nothing?")** | The flat-fee `BaselinePricing` is already implemented as the no-mechanism comparator. Reporting the welfare delta vs flat-fee for each menu option is the cleanest "what is this mechanism worth?" framing — closely parallel to Roughgarden's "EIP-1559 vs first-price baseline" comparison. The `mechanism-welfare-impact` analysis confirms `baseline_flat_fee` is byte-identical between mechanism variants, so it's a sound zero. | Low (~½ day) — the suite runs already include `baseline_flat_fee`; the comparator table just needs to be assembled | **STRONGLY RECOMMENDED** — turns absolute welfare numbers into welfare *gains*, which is the actual reader question |
| D2 | **Comparator-baseline against EIP-1559 deployment data (not just simulated)** | The empirical EIP-1559 deployment papers (Liu et al. 2022, "Empirical Analysis of EIP-1559"; Reijsbergen et al. 2021) report real-world wait-time and base-fee dynamics that act as a sanity-check on the simulator's single-lane numbers. If the simulator's single-lane EIP-1559 outputs are in the same shape as deployed Ethereum's, that's a strong externality-of-simulation argument. | Medium (~1–2 days) — fetch one or two well-known plots from the Liu/Reijsbergen papers, qualitatively compare to the phase-2 simulator's single-lane time-series output | **RECOMMENDED** — single-lane is held as control evidence anyway; a quick "looks like deployed Ethereum" check pays off in reviewer trust at low cost |
| D3 | **Adversarial-case stress tests for the published menu items** | `paper_like_mispriced` already exists as one such test (urgency-inverted demand under aggressive fee budgets). One additional "what would break this conclusion?" test per menu item — e.g., congestion spikes calibrated to be near the worst-case the mechanism survives — turns the evidence base from "shows the mechanism works at typical load" to "shows the mechanism's failure modes are characterised." | Medium (~1 day per stress test; ~3–5 tests) — leverage `paper_like_mispriced` as the pattern; one stress per menu item is sufficient | **RECOMMENDED** — `PROJECT.md` Out-of-Scope excludes *strategic-bidder* adversarial modelling, which is the more expensive cousin; demand-profile stress tests are cheap and answer most of the same "robustness" intuition |
| D4 | **Mechanism-independence claim with explicit boundary** | The `mechanism-welfare-impact-2026-05-14.md` data already demonstrates this: the top two menu arms (un-reserved priority-only, un-reserved both-dynamic) are mechanism-robust (median \|Δ%\| 15%, 0 sign flips) between Family A and Family B. RB-reserved and partitioned arms gain ~30% under Family B with `x4_rb_quarter` corner-flips. Reporting this as a "boundary-of-applicability" result — the un-reserved arms hold across cadence choices; the RB-reserved arms hold except at extreme stress — is publication-grade. | Low (~½ day) — data exists; needs lifting from `mechanism-welfare-impact-2026-05-14.md` into a CIP-citeable robustness table | **STRONGLY RECOMMENDED** — converts the otherwise-painful "we changed the mechanism mid-publication" narrative into a strength |
| D5 | **Independent reproductions / formal-methods backing (light touch)** | The CLAUDE.md numeric-representation contract (integer/rational/u128 in simulation-affecting state; libm where bit-stability matters) is *almost* a formal-methods argument — the controller trajectory is reconstructible from the canonical chain by anyone with the spec. Stating this explicitly ("the controller's full trajectory is a pure function of the canonical chain and pricing settings; any third party with the spec can re-derive it byte-identically") is a quasi-formal claim with real teeth. | Very low (~½ day) — `family-b-decision-2026-05-14.md` "Publication framing" section already drafts this | **STRONGLY RECOMMENDED** — chain-derivation (Family B) earns this property by construction; not surfacing it loses the strongest single epistemic claim available |
| D6 | **Pool-number sensitivity test** | `PROJECT.md` lists this as Active: 33-job smoke × {100, 150 pools}. The point is to license the choice of `topology-realistic-100.yaml` against the obvious-reviewer-question "shouldn't this match Cardano's actual ~600 active pools?". A null result (100 ≈ 150 on welfare metrics) is the publication-citable "topology is not load-bearing in the relevant range" disclosure. | Medium (~1 day compute; recipe established by `topology-realistic-100` + `generate-realistic-100-topology.py`) | **STRONGLY RECOMMENDED** — preempts the single most-likely reviewer objection at low cost; replaces the deferred 600-pool migration |
| D7 | **Run-length / steady-state validation** | `PROJECT.md` Active: verify 2000 slots suffices for steady-state under each menu item. The 2000-slot default is calibrated under congested-paper-like demand; menu items with different equilibrium time-constants (e.g., RB-reserved priority-only at low priority demand) may not have reached steady state in 2000 slots. A diagnostic plot (`quote_per_byte` time-series for each menu item, looking for a clear plateau) resolves the question cheaply. | Low (~½ day) — the time-series outputs are already collected; the analysis is "do these flatten by slot 2000?" | **RECOMMENDED** — un-resolved, it is a hidden interpretation risk for any magnitude claim; resolved (with disclosure for any menu item that needs longer runs), it is a cheap rigour win |
| D8 | **Per-option "this is when you would not pick this"** | A menu CIP whose options each say "pick me" without anti-recommendation is doing reviewer-burden offloading. Picking each menu item's anti-use-case explicitly (e.g., "un-reserved priority-only is dominated by RB-reserved when anti-bribery is a hard constraint") demonstrates that the author understands the option space well enough to navigate it. | Low (~½ day) — derived directly from the trade-off statements in Table Stakes #9 by inversion | **RECOMMENDED** — pairs cleanly with #9; together they form the "menu-of-options" framing the user has chosen |

## Anti-Features

Features that *look* like rigour, are easy to ship, and crowd out the real work. Naming these explicitly so the evidence package does not accidentally bias toward shiny-and-cheap over hard-and-load-bearing.

| # | Anti-feature | Why it's tempting | Why it doesn't help | What to do instead |
|---|--------------|-------------------|--------------------|--------------------|
| A1 | **Summary statistics without uncertainty quantification** | Cheap to compute; tables look clean. A single number per (mechanism × demand profile) cell fits in a slide. | Roughgarden 2021's empirical-EIP-1559 critique is that the original EIP-1559 community survey cited "lack of formal specification or proof that people can independently evaluate" as chief risk — point estimates without bounds are exactly that gap. The mechanism-welfare-impact data already shows 4 of 33 jobs flip sign between cadence variants; a point estimate that happens to fall on one side of zero gives a *worse* impression than median + IQR with the flip disclosed. | Median + IQR (or interquartile) across the N seeds; explicit sign-flip annotation where present. `validity-threats.md` already prescribes "3-seed median, IQR" framing rather than 95% CI. |
| A2 | **Single-seed claims presented as conclusive** | The 33-job sundaeswap smoke at seed=1 is the canonical Family B characterisation, and reproducing the 33-job set at multiple seeds is a real compute investment. A single seed *looks* fine when the result is dramatic. | The pre-Family-B `family-b-results-table-2026-05-14.md` numbers and the post-bug-#1 `mechanism-welfare-impact-2026-05-14.md` numbers are both seed=1 only. Any conclusion drawn from these is a "single-seed claim" by definition. A reviewer who has read Williams 2024 or any blockchain-simulator reproducibility checklist will flag this. | `PROJECT.md` Active "Multi-seed variance bands" deliverable. N seeds calibrated by run cost; the canonical menu-item jobs (not all 33) are the priority targets. |
| A3 | **Parameter sweep without baseline contrast** | `phase-2-eip1559-robustness` sweeps D ∈ {4, 8, 16} × target ∈ {0.25, 0.5, 0.75}; the results table shows welfare numbers across the 5-job diagonal. Without a "and here is flat-fee under matched demand" row, the sweep proves only that the mechanism is internally smooth — not that the smoothness is *worth* anything. | Reviewers reading the sweep table look for "is this better than nothing?" first and "is parameter X load-bearing?" second. Without the baseline row, both questions stay open even after the sweep is read. | Every sweep table includes a `baseline_flat_fee` row (or column). The data is already there from `baseline_flat_fee` job entries; assembly is the cost. |
| A4 | **"We tested it" assertions without listing the tests** | A sentence like "the mechanism has been extensively tested across demand profiles and topologies" is template-CIP language that costs nothing to write. | The CIP-1 Rationale standard ("evidence of consensus … significant objections") makes this kind of language reviewer-bait. Specifically: the 19 phase-2 suites under `parameters/phase-2-sweep/suites/` already each have a question + scope + jobs definition; *naming* them in the CIP is the difference between "trust me" and "verify me." | Per-claim mapping table (Table Stakes #1) does this concretely. A separate "tests run" appendix can list the 7 goldens-pinned suites + the 12 demand-regime suites with one-line question-and-answer for each. |
| A5 | **Audit-trail of fixed bugs that hides what the un-fixed bugs hid** | `REVIEW.md` Fix Status is honest about which findings were applied vs deferred (WR-2, WR-7), but the *interpretive* question — "did the bugs that were fixed change conclusions?" — is left implicit. Same for the chain-derived refactor: `mechanism-welfare-impact-2026-05-14.md` honestly shows that 4 of 33 jobs flipped sign between accumulator and chain-derived, but the framing has to make explicit "we made a mechanism choice mid-publication; here are the cells affected." | A reviewer who reads Family B framing as "we found a bug and fixed it" will mistrust the suspended judgement on the cells where the sign actually flipped. Conversely, a reviewer who reads it as "we discovered an unintentional implementation artifact, characterised its impact across 33 jobs, and chose the spec-faithful variant for publication" gets the right epistemic frame. | The `family-b-decision-2026-05-14.md` "Publication framing" already nails this. The CIP needs to cite the explicit cells that flipped (`eip1559_d4_t50_w32`, `eip1559_d8_t25_w32`, `rb_reserved_x4_rb_quarter`, `partitioned_x4_rb_quarter`) rather than hand-wave that "some cells changed." |
| A6 | **"Sensitivity sweep" that only varies one knob at a time and reports no interaction** | OFAT (one-factor-at-a-time) sweeps are cheap and produce one bar chart per knob. They look like sensitivity analysis. | The Ten Tusscher et al. ABM sensitivity-analysis literature (Lorscheid 2012, Broeke 2016) flags OFAT as the canonical failure mode: it cannot detect interactions, which are exactly where calibration fragility lives. The `multiplier_floor × RB-capacity-reduction` interaction surfaced by Family B's `x4_rb_quarter` cell is precisely such an interaction; an OFAT sweep over `multiplier_floor` alone (e.g., `phase-2-priority-only-rb-reserved`'s 3-job sweep) would *not* have surfaced it. | Two-factor sweep at the corners where interactions are theoretically plausible. `multiplier_floor × RB-capacity-reduction` is the obvious one; the existing `phase-2-rb-scarcity` suite is structured for exactly this. Explicit interaction-flagging in the results writeup. |
| A7 | **Disclosure section that's actually a "we are aware of limitations" hedge** | "Of course, this simulator does not perfectly model adversarial behaviour …" reads as appropriate humility. It's also content-free. | Roughgarden 2021's framing of EIP-1559 in terms of named impossibility properties (DSIC under demand spike, MMIC, OCA-proofness) is the reverse: each limitation is a *specific* property the mechanism is *known not* to deliver, with a *specific* condition. That's actionable for downstream reviewers; "we don't model X" is not. | Per-limitation: name the property not delivered; cite the condition under which the limitation fires; name what would resolve it. The `family-b-decision-2026-05-14.md` "Follow-on work" section is the template. |
| A8 | **Citing the realism audit as if it audited everything** | `cardano-realism-audit.md` covers calibration choices and modeling assumptions. Citing it as "the simulator has been audited" elides what was *not* audited. | `PROJECT.md` Out-of-Scope #3 explicitly excludes re-auditing the upstream Leios simulator code paths. The audit is for the *phase-2 rebuild*, not the substrate. Citing the audit as a "the whole simulator is audited" claim is a misuse. | Disclosure paragraph: phase-2 rebuild's audit + the inherited-simulator-substrate's disclosed-limitation status are distinct epistemic claims. The CIP's reproducibility-recipe section can be honest: "phase-2 pricing kernel and metrics layer are integer/rational throughout; the upstream Leios substrate (lottery, propagation, voting) is inherited from `main` and retains f64 in non-pricing paths." |

                                      Active for 4 unanchored knobs

D4  (mechanism-independence)        ← mechanism-welfare-impact-2026-05-14.md complete

## MVP Recommendation

The minimum viable evidence package for the CIP cites:

1. **Table Stakes #1** (claim → job mapping). Without this the menu CIP cannot be reviewed at all.
2. **Table Stakes #2** (variance bands at least for the headline menu-item claims). Multi-seed re-runs of the canonical jobs are the load-bearing rigor investment.
3. **Table Stakes #4** + **Table Stakes #5** (calibration provenance + disclosure consolidation). The `cardano-realism-audit.md` rewrite + a single disclosure section together.
4. **Table Stakes #7** (realism-risks register). The user's own framing.
5. **Differentiator D1** (vs flat-fee baseline). Converts welfare numbers into welfare *gains* — answers the actual reader question at near-zero cost.
6. **Differentiator D4** (mechanism-independence claim with explicit boundary). Turns the Family B narrative from "we changed mechanisms" into the strongest single epistemic claim in the package.
7. **Differentiator D6** (pool-number sensitivity test). Preempts the most-likely reviewer objection at low cost.

Defer (justifying in the CIP as out-of-scope-but-disclosed):
- **D2 (deployed-EIP-1559 baseline)** — pleasant-to-have, not load-bearing. The flat-fee counterfactual is the load-bearing comparator.
- **D3 (extra stress tests beyond `paper_like_mispriced`)** — one stress per menu item is *recommended*; cutting all of them is acceptable if compute is tight.
- **D5 (independent reproductions / formal-methods backing)** — `D5` is mostly free given chain-derivation; the *light-touch* version (the publication-framing paragraph) is in scope, the *full* version (third-party reproduction) is out.

Out of scope per `PROJECT.md`:
- Adversarial/strategic-bidder modelling. The CIP discloses as future work.
- Cross-architecture CI verification. Disclosed; intra-arch determinism via pinned hashes is sufficient.
- Re-auditing upstream Leios simulator substrate. Disclosed as limitation, not re-audit target.
