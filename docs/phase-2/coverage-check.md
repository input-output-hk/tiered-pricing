# Coverage Check

**Status:** v1 — Phase 2 skeleton committable for Phase 3 entry (per COV-06); Phase 3 will populate BACKED cells as cheap-test results arrive
**Scope:** Per-claim coverage map for the Cardano Improvement Proposal (CIP) responding to CPS-0023 ("Urgency Signaling"). Each row is one `(claim, menu-option)` pair; rows are atomic and greppable by claim, menu-option, status, or related-RSK reference.
**Identifier convention:** `CLM-NN` (append-only; never renumbered) per Leios `docs/ImpactAnalysis.md` precedent. Cross-references to the realism-risks register use the `RSK-NN` identifiers from `docs/phase-2/realism-risks-register.md` directly.
**Verdict vocabulary:** `BACKED` / `WEAK` / `UNBACKED` / `OUT-OF-SCOPE`. Different vocabulary from the register's `LIVE` / `DORMANT` / `MITIGATED` / `DISCLOSED`. `BACKED` requires the hash-diversity gate (see "Hash-diversity gate" section below).

## Reading guide

This file is the single source of truth for menu-item-trade-off claim coverage. Each row maps one `(claim, menu-option)` pair to a specific backing simulator job (or to a by-construction citation, or to a mainnet calibration triple). Per-claim trust ratings on the upstream simulator substrate live in `docs/phase-2/validity-threats.md`; this file does not duplicate those — it absorbs them into per-row `confidence-method` cells.

The four CIP menu options (`priority-only-RB-reserved`, `priority-only-un-reserved`, `both-dynamic-partitioned`, `both-dynamic-un-partitioned`) are evaluated against the single-lane `single-lane-EIP-1559-control` baseline. The control is retained as control evidence per `.planning/PROJECT.md`; it is not promoted as a CIP menu item.

The shape follows the Requirements Traceability Matrix (RTM) idiom — denormalised one row per `(claim, menu-option)` pair so that a reader can grep by claim text, menu option, status, or `RSK-NN` reference and find every matching row. With ~10 distinct claim types × 5 menu options (and a few claims that do not apply to every option), the table lands in the 20–50 row range.

## Hash-diversity gate (COV-05; Phase 3 enforces)

Per `.planning/REQUIREMENTS.md` REQ-COV-05 and CONTEXT.md D-19 (strict rule):

> A row may only be cited as `BACKED` when the distinct `pricing_event_stream.sha256` count equals the seeds-cited value. Rows whose seed-set collapses to fewer distinct hashes are downgraded to `WEAK` with annotation, or are re-run with different seed values.

The gate is enforced in Phase 3 after multi-seed runs land. Phase 2 documents the gate semantics but does not apply it (`BACKED` promotions are Phase 3 work). The full-sweep evidence base (`.planning/family-b-full-sweep-analysis-2026-05-14.md`) already shows 468 of 468 (job, seed) pairs in the 3-seed sweep produced 3 distinct stream hashes per cell, so the gate is expected to pass everywhere it is applied; the policy decision recorded in `RSK-hash-diversity-policy` is to apply the strict form of the gate.

- **`EXP-<slug>` forward references** in the `claim` cell name the Phase 3 / Phase 4 work item that will land evidence for that row, per CONTEXT.md Claude's Discretion §"EXP-NN forward references".

### Headline finding for Cardano Improvement Proposal (CIP) author

At the multiplier_floor = 4 calibration under sundaeswap_moderate demand:
- **Un-reserved menu arms (priority-only and both-dynamic) materially outperform single-lane EIP-1559** with tight 95% CIs.
- **RB-reserved menu arms (priority-only and both-dynamic) underperform single-lane EIP-1559** with tight 95% CIs — Phase 3 statistically defends a finding that contradicts the Phase 1 / Phase 2 single-seed framing.
- The cross-arm duplicate-job artefact (partitioned ≡ rb-reserved welfare under this calibration) is confirmed at N=20.
- The multiplier_floor = 4 calibration is **regime-dependent**: at floor = 16 the mechanism rejects most demand and welfare characteristics invert (TEST-07a).

This finding is load-bearing for the CIP's menu-option recommendation and should anchor the Limitations / Evidence sections of the CIP.
