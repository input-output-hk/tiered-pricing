## Reading guide

This document is the one-page Overview, Design concepts, Details (ODD) index for the phase-2 dynamic-pricing simulator. It serves three roles. **First**, it is a citation target: the Cardano Improvement Proposal (CIP) produced by Phase 5 of this project links here by repository (repo) Uniform Resource Locator (URL) rather than embedding methodology prose inline. **Second**, it is a navigation aid: the §"Index" table below maps each of the seven canonical ODD elements (Purpose, State variables, Process overview, Design concepts, Initialisation, Input data, Submodels) to the in-repo file or section that contains the authoritative content for that element. **Third**, it is onboarding documentation: the §"Worked example" near the end of the document traces a single canonical (job, seed) pair end-to-end through the seven elements so a new contributor can read the document in isolation and understand how the simulator's pieces fit together.

The methodology overview deliberately does **not** duplicate prose from the linked files. The four sibling documents in [`docs/phase-2/`](.) carry the substantive content and should be cited alongside this overview when the CIP needs detail beyond the one-line summaries here:

### Purpose

Phase-2 evaluates four Cardano Improvement Proposal (CIP) dynamic-pricing menu options — ranking-block-reserved priority-only-static (RB-reserved priority-only-static), un-reserved priority-only-static, both-dynamic partitioned, and both-dynamic un-partitioned — against a single-lane Ethereum Improvement Proposal 1559 (EIP-1559) control on welfare metrics under a Family-B chain-derived controller (see [`docs/phase-2/mechanism-design.md`](mechanism-design.md) §"Live mechanisms" for the spec, and [`.planning/research/SUMMARY.md`](../../.planning/research/SUMMARY.md) for the upstream Cardano Problem Statement 23 ("Urgency Signaling") (CPS-0023) problem framing this work responds to). The primary per-(job, seed) welfare metric is `retained_value`, totalled across priority and standard lanes (an f64-reporting value, never read back into a simulation decision; see [`CLAUDE.md`](../../CLAUDE.md) §"Numeric representation contract" for the f64-reporting convention). The four menu options share the underlying linear-Leios protocol; the differences are localised to lane structure, ranking block (RB) partitioning, and which lane's quote is dynamic. No new on-chain primitive is introduced; the existing transaction fee field is reinterpreted as a maximum-fee envelope (see [`mechanism-design.md`](mechanism-design.md) §"EIP-1559 maximum-fee semantics" for the reinterpretation, and the realism-risks register entry `RSK-fee-as-maxFee-envelope` in [`realism-risks-register.md`](realism-risks-register.md) for the hard external dependency on the refund-handling CIP). The CIP that downstream Phase 5 produces cites this Purpose and its evidence base, not the original phase-1 tiered-pricing framing.

### State variables

The simulator's primary state variables are the canonical-chain types defined in [`sim-rs/sim-core/src/model.rs`](../../sim-rs/sim-core/src/model.rs) and summarised in [`CLAUDE.md`](../../CLAUDE.md) §"Mechanism abstractions": `Transaction` (carries `posted_lane: Lane`, `bytes: usize`, `max_fee_lovelace: u64`, and a per-component `urgency: f64` read only by the actor lane-choice math), `LinearRankingBlock` (the ranking block (RB), which carries `derived_quote: PerLaneQuote` and `window_aggregate: WindowAggregate` as header fields), `LinearEndorserBlock` (the endorser block (EB), which carries `partition_activated: bool` so endorsers and producers agree on whether the RB-reserved priority partition was triggered for that block), and the per-node `Mempool` whose byte cap is `2 × eb_referenced_txs_max_size_bytes`. The `PerLaneQuote { standard: u64, priority: u64 }` and `WindowAggregate { standard_sum_bytes, standard_sum_capacity, priority_sum_bytes, priority_sum_capacity, blocks_in_window }` (all u128 sums) are the controller's chain-derived state — they live on the canonical chain, not in per-node memory, so they are reorg-safe by construction (see [`docs/phase-2/mechanism-design.md`](mechanism-design.md) §"Chain-derived controller" for the design rationale). Every state variable that influences which transaction lands in which block is integer- or u128-typed; the only `f64` allowed on hot paths is `urgency`, and it routes through `libm::pow` + `libm::round` into `i128` lovelace before any comparison so lane choice stays bit-deterministic across architectures.

### Process overview

One simulation step corresponds to one Cardano slot. At each slot the protocol cycle runs in this order: (1) the slot lottery (`sim-rs/sim-core/src/sim/lottery.rs`) elects, per stake-weighted Verifiable Random Function (VRF), which nodes may produce a ranking block (RB) and which may produce votes (see [`CLAUDE.md`](../../CLAUDE.md) §"Mechanism abstractions" for the lottery semantics); (2) elected producers consult their local mempool and build candidate blocks via [`sim-rs/sim-core/src/sim/linear_leios.rs`](../../sim-rs/sim-core/src/sim/linear_leios.rs)'s `try_generate_rb` and `select_eb_with_partition` functions; (3) before publication, the producer computes the new block's `derived_quote` via `compute_chain_derived_quote_for_child_of`, which calls the active `PricingBackend::compute_derived_quote` as a pure function of the parent block's chain-derived state plus the parent's emitted samples; (4) endorsement-stage producers validate any referenced endorser block (EB) via `eb_endorsement_valid` — an EB whose transactions no longer pass the current quote check is refused endorsement; (5) admitted transactions go through the mempool gate ([`sim-rs/sim-core/src/sim/mempool_gate.rs`](../../sim-rs/sim-core/src/sim/mempool_gate.rs)) which enforces `minFeeB + quote × bytes ≤ max_fee_lovelace`; (6) included transactions emit `Event::TXIncluded` with `actual_fee_lovelace` and `refund_lovelace`. The controller advances exactly once per canonical block (per Ethereum Improvement Proposal 1559 (EIP-1559) cadence), never twice — the pre-2026-05-14 accumulator implementation was a chain-derivation bug, not the spec (see [`CLAUDE.md`](../../CLAUDE.md) §"Mechanism choice and audit trail" for the audit trail).

### Design concepts

Five design concepts are load-bearing across phase-2. **First, chain-derivation:** the controller is stateless at the node level — each `LinearRankingBlock` carries its own `derived_quote` as a pure function of its parent's chain-derived state and the parent's emitted samples (see [`docs/phase-2/mechanism-design.md`](mechanism-design.md) §"Chain-derived controller"). This makes the design reorg-safe by construction: orphan blocks from slot battles carry their own `derived_quote` which is discarded when the block is discarded, so contamination of the canonical pricing trajectory by short forks is impossible (resolving the writeup-risk-1 (WR-1) reorg-contamination concern from [`.planning/REVIEW.md`](../../.planning/REVIEW.md)). **Second, integer/rational discipline:** simulation-affecting state (admission, eviction, fee charging, controller coefficient, mempool tracking, lane choice) is integer or u128 throughout; the EIP-1559 update rule runs in u128 rationals (`aggregateUtil = Σ bytes / Σ capacity`, `target = (num, den)`, `D` integer, clamped step on `quote_per_byte`); plain `f64` is only allowed in the reporting layer (`retained_value`, `net_utility`, time-series ratios) and never feeds back into a simulation decision (see [`CLAUDE.md`](../../CLAUDE.md) §"Numeric representation contract"). **Third, multiplier-floor invariant:** `c_priority ≥ multiplier_floor × c_standard` is enforced post-update on `quote_per_byte` with u128 intermediates and ceiling division; the floor is the price-discrimination guarantee that distinguishes the priority lane from the standard lane. **Fourth, intra-architecture determinism:** the M2/M3 unit-test goldens and the M5 suite-level pricing-event-stream SHA256 goldens reproduce bit-identically on the same architecture (the development machine is `x86_64` / `glibc`); cross-architecture verification is documented as not-yet-built and is out of phase-2 scope. **Fifth, deterministic event-stream invariants:** the pricing event stream over `TXIncluded` and `TXEvictedQuoteDrift` events is the contract that golden hashes pin (see [`.planning/codebase/CONVENTIONS.md`](../../.planning/codebase/CONVENTIONS.md) §"Numeric representation contract" and `sim-rs/sim-cli/tests/determinism.rs`).

### Initialisation

A simulation run starts from configuration deserialised by [`sim-rs/sim-core/src/config.rs`](../../sim-rs/sim-core/src/config.rs) from a layered Yet Another Markup Language (YAML) hierarchy: a base [`parameters/config.default.yaml`](../../sim-rs/sim-cli/parameters/config.default.yaml) embedded via `include_str!`, then the per-suite [`protocol-base.yaml`](../../sim-rs/parameters/phase-2-sweep/protocol-base.yaml) (Phase-2 protocol knobs: `min-fee-a = 44`, `min-fee-b = 155381`, `rb-generation-probability = 0.05`, `eb-referenced-txs-max-size-bytes = 12000000`, the chain-derived controller parameters), then the topology YAML ([`topology-realistic-100.yaml`](../../sim-rs/parameters/phase-2-sweep/topology-realistic-100.yaml) is the default for all phase-2 suites since 2026-05-13 — 100 nodes, mass-stratified mainnet stakes), then the per-job demand YAML ([`demand/sundaeswap_moderate.yaml`](../../sim-rs/parameters/phase-2-sweep/demand/sundaeswap_moderate.yaml) and the four `paper_like_*` variants are the actor-component profiles), then the per-job pricing YAML ([`pricing/*.yaml`](../../sim-rs/parameters/phase-2-sweep/pricing/) — 19 files across single-lane EIP-1559, RB-reserved priority-only-static, un-reserved priority-only-static, and both-dynamic variants at multiplier-floor values 4, 8, 16). The simulator's initial state is then deterministic in the seed: an empty per-node `Mempool`, an empty canonical chain, a `PerLaneQuote` initialised at each pricing YAML's `initial-quote-per-byte` (typically `minFeeA = 44` for standard, `multiplier_floor × minFeeA` for priority), an empty `WindowAggregate`, and a slot-lottery state seeded by the `(seed, slot)` pair. All four pieces of configuration are inputs to the run's reproducibility — changing any flips the M5 suite-level golden SHA256.

### Input data

Phase-2 has three empirical anchors. **First, the Cardano mainnet stake distribution:** the [`topology-realistic-100.yaml`](../../sim-rs/parameters/phase-2-sweep/topology-realistic-100.yaml) header documents a mass-stratified downsample of the 1,510 Cardano mainnet stake pools with active stake ≥ 1k ADA (Cardano on-chain state at epoch 582, retrieved 2026-05-14), rescaled linearly to total = 3 × 10^10 lovelace (top-1 stake share = 1.97%, Nakamoto coefficient = 35, Gini = 0.253). The reproduction recipe is in [`sim-rs/scripts/generate-realistic-100-topology.py`](../../sim-rs/scripts/generate-realistic-100-topology.py). **Second, the Cardano mainnet `activeSlotsCoeff`:** the protocol-baseline value `rb-generation-probability = 0.05` is the mainnet value at the same retrieval date; the calibration-fix postmortem at [`docs/phase-2/calibration-fix-postmortem.md`](calibration-fix-postmortem.md) records the trace from an earlier `rb-prob = 1.0` framing to the corrected mainnet value and explains why `rb-prob ≥ ~0.077` re-introduces the endorsement-window bug. **Third, the demand profiles:** the `sundaeswap_moderate.yaml` profile is calibrated against the publicly reported SundaeSwap mainnet launch event of January 2022 (the canonical Cardano congestion event); the `paper_like_*` profiles are calibrated to the Decentralised Exchange (DEX) / large-transaction throughput characteristics reported in upstream Leios research papers. The cardano-realism audit at [`docs/phase-2/cardano-realism-audit.md`](cardano-realism-audit.md) §"What lines up with mainnet" carries the full `(value, source, date-retrieved)` triple inventory.

### Submodels

The phase-2 simulator decomposes into four named submodels, each with a well-defined interface. **`PricingBackend`** ([`sim-rs/sim-core/src/tx_pricing/mod.rs`](../../sim-rs/sim-core/src/tx_pricing/mod.rs)) is the trait that abstracts the controller policy: implementations include `BaselinePricing` (flat-fee `c = 1`), `Eip1559Pricing` (single-lane EIP-1559 step), and `TwoLanePricing` with four `TwoLaneVariant` arms covering the spec's matrix (RB-reserved priority-only-static, un-reserved priority-only-static, both-dynamic partitioned, both-dynamic un-partitioned); the trait method `compute_derived_quote` is pure (no `&mut self`, no node-local state). **`MempoolGate`** ([`sim-rs/sim-core/src/sim/mempool_gate.rs`](../../sim-rs/sim-core/src/sim/mempool_gate.rs)) is the sole byte-cap authority: owns admission (`minFeeB + quote × bytes ≤ max_fee_lovelace` and not over byte cap), revalidation on quote change (evicts transactions whose lane's quote has risen above their `max_fee_lovelace`), inclusion charging (`actual_fee = minFeeB + quote(served_lane) × bytes`, `refund = max_fee − actual_fee`), and reject-only-on-full semantics (no eviction to make room for new arrivals). **`ActorComponent`** ([`sim-rs/sim-core/src/tx_actors.rs`](../../sim-rs/sim-core/src/tx_actors.rs)) is the demand-side actor model: weighted value-urgency sampling produces a stream of transactions; `MaxFeePolicy::ScaledOverLaneQuote { numerator, denominator }` produces `max_fee_lovelace` deterministically (default `{4, 1}`, i.e. 4× quote-drift headroom); `LanePolicy::UtilityMaximising` picks `posted_lane` by maximising expected utility through `libm::pow` + `libm::round` into i128 lovelace; per-(component, lane) `LatencyEstimator` keeps an Exponentially-Weighted Moving Average (EMA) of observed inclusion latencies. **`WindowAggregate`** update ([`sim-rs/sim-core/src/tx_pricing/window.rs`](../../sim-rs/sim-core/src/tx_pricing/window.rs)) implements the rolling capacity-weighted utilisation sums (u128 throughout) as two pure functions: `aggregate_from_chain` for cold-start aggregation and `update_aggregate` for the per-block incremental step. The chain-derived production path in [`sim/linear_leios.rs`](../../sim-rs/sim-core/src/sim/linear_leios.rs) wires these submodels together; the only seam between the simulator and the pricing kernel is the `ChainView` trait (a read-only walk of the canonical chain).
