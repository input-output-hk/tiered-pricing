# [Experimentation](https://github.com/input-output-hk/tiered-pricing/issues/54)

Completed the first round of comparisons across the phase-2 candidate mechanisms in the abstract simulator. [Diff here](https://github.com/input-output-hk/tiered-pricing/pull/75/changes); [preliminary report here](https://github.com/input-output-hk/tiered-pricing/blob/cip-urgency-signalling/docs/phase-2/preliminary-experiment-report.md).

The experiments compare the static flat-fee control and single-lane EIP-1559 baseline against the priority mechanisms under active consideration:

- Priority-only dynamic pricing vs both-standard-and-priority dynamic pricing
- Open priority-first selection vs ledger-enforceable RB reservation for priority transactions
- Instant priority-reservation utilisation signals vs windowed priority signals, with window sizes 3, 5, 10, and 20
- Ten seeded runs per mechanism under an identical severe-congestion workload, so that differences are attributable to the mechanism rather than to different demand samples

Simulator and metrics work this fortnight included:

- Added the committed mechanism sweep harness and phase-2 sweep manifest for reproducible candidate-design x seed comparisons
- Added the windowed priority price signal variants, and fixed the standard-lane signal to include ex-units
- Added/refined refund semantics so that the priority premium can depend on where a transaction is actually included, not just which lane it entered
- Tightened the price stability metrics, including price shocks and true oscillation cycles
- Improved demand-unit accounting for inclusion, latency, retained/lost/unresolved value, and fairness/starvation
- Fixed config issues found while exercising the sweep, including the flat-fee control and the `both-dynamic-open` priority delivery mode
- Updated the dashboard/preprocessor to support sweep comparison, experiment/seed switching, value and fate breakdowns, latency by lane or urgency class, fairness, and price oscillation markers

The headline result is that a priority lane materially improves outcomes for urgent transactions. Across ten seeded runs, the best aggregate row reduces urgent mean latency from 2.91 to 2.39 blocks (~18%) and improves urgent retained value from 44.32% to 51.65% (+7.33 percentage points, ~16.5% relative). The best reserved variant is very close at 51.55% urgent retained value, so the current recommendation is to prefer a reserved design (`priority-only-reserved window 5` or `both-dynamic-reserved window 5`) because it keeps most of the performance benefit while enabling ledger enforcement.

# [Prototype](https://github.com/input-output-hk/tiered-pricing/issues/55)

**Trajectory.** The prototype is built bottom-up — ledger → mempool → consensus → testing → node integration — so each layer can be exercised before the next depends on it. The familiarisation tasks ([read the paper #61](https://github.com/input-output-hk/tiered-pricing/issues/61), [phase-1 writeup #62](https://github.com/input-output-hk/tiered-pricing/issues/62), [linear-Leios prototypes #63](https://github.com/input-output-hk/tiered-pricing/issues/63)) are done. This fortnight moved from theory to code: the [ledger changes #66](https://github.com/input-output-hk/tiered-pricing/issues/66) are implemented, the [mempool changes #68](https://github.com/input-output-hk/tiered-pricing/issues/68) are implemented and unit-tested, and the whole stack has been run and pressure-tested end-to-end on a private devnet. The work spans six repos: `cardano-ledger` (`nicolas/dynamic-pricing`, the ledger rules), `ouroboros-consensus` (`leios-prototype`, the 2-lane mempool), `cardano-api` / `cardano-cli` / `cardano-node`'s tx-generator (a Dijkstra transaction path and lane feeder), and `ouroboros-leios` (the `proto-devnet` harness). Everything builds against Nicolas' nix devshells; all commits are local (no push).

## Ledger changes ([#66](https://github.com/input-output-hk/tiered-pricing/issues/66))

The spec is implemented as atomic commits, mapping onto the issue's three sub-tasks:

**1. Dynamic fee coefficient ([#70](https://github.com/input-output-hk/tiered-pricing/issues/70)).** New `Cardano.Ledger.DynamicPricing` domain (`InclusionStrategy`, `Pricing`, `State`) carrying the per-lane price coefficient, exposed via an `EraPricing` era-family that follows the `EraGov`/`EraCertState` pattern. `UTxOState` gains a family-typed `utxosPricing` field — `NoPricing` for Shelley→Conway, `DynamicPricing` for Dijkstra, seeded by the Conway→Dijkstra translation. On the transaction side, the Dijkstra `TxBody` gains `dtbInclusion` (CBOR key 27, default `Optimistic` omitted), `dtbFeeRefundAccount` (key 28), and `dtbBidFee` (the fee field read as a price cap).

**2. Fee-validation rule ([#72](https://github.com/input-output-hk/tiered-pricing/issues/72)).** The UTXO rule gates each tx on its bid: U1 replaces the plain min-fee premise with `quoteFor` (`BidBelowQuote` on failure), and U2 records per-strategy usage. The fee split on `feeRefundAccount` keeps the base (protocol minimum) in the fee pot, routes the premium (`quote − base`) to the treasury via the donation pot, and records the refund (`bid − quote`) as pending. The LEDGER rule then `flushPendingRefunds` — crediting registered accounts after each tx; refunds to unregistered accounts stay pending (*waiting for a logic to handle it from Polina*).

**3. Fee-update rule ([#71](https://github.com/input-output-hk/tiered-pricing/issues/71)).** At the end of every block the prices for the next block are recomputed from how full each lane was — Will's EIP-1559 controller nudges a lane's price up when it runs hot and down when it runs cold, toward a target utilisation. (Implemented as the BBODY rule plus a `DynamicPricing.Controller` module, covered by its own tests.)

*In progress:* a clean-up pass splitting the dynamic-pricing state into focused domain modules, and getting the full ledger test suite green again after the new pricing field was threaded through it.

## Mempool changes ([#68](https://github.com/input-output-hk/tiered-pricing/issues/68))

Implemented a naive, first-pass version of the protocol in `ouroboros-consensus` (`leios-prototype` / linear-Leios) — enough to exercise the two-lane behaviour end-to-end, with the more refined policies (eviction, replace-by-fee, cross-lane dependencies) deliberately left for later. The mempool already re-validates every tx through the ledger rules at each new tip, so evicting repriced txs comes essentially for free once the `bidFee ≥ quote` gate lives on the reapply path. Added a new `Ouroboros.Consensus.Mempool.Lanes` module holding the pure lane policy: two lanes (Urgent / Optimistic) chosen by a new `txInclusionLane` method on `LedgerSupportsMempool` (default `UrgentLane`, so other eras are unchanged), per-lane capacities, and a forge-selection rule that preserves the logical ledger order (`Urgent* ++ Optimistic*`) — the RB takes the urgent prefix plus optimistic filler only when no urgent tx is left behind, the EB is optimistic-only. Wired through admission, revalidation and forge (mempool API, capacity, common impl, update path, HardFork combinator, NodeKernel). Forge observability comes via a `ForgeLaneSummary` returned by `snapshotTakeForForge` and traced on the existing `leiosKernelTracer`, so no new shared trace constructors were needed. Covered by deterministic unit tests plus QuickCheck properties — all green (`consensus-test -p /Lanes/`: 13 passed).

## Transaction tooling (cardano-api / cardano-cli / cardano-node)

A tiered-pricing transaction couldn't be built or submitted with the existing tooling, so this also required changes up the node stack. `cardano-api` gains a public `TxInclusion` surface — a `TxInclusion era` GADT (`TxInclusionNone`, or `TxInclusion Inclusion` for Dijkstra), a `txInclusion` field on `TxBodyContent`, `setTxInclusion`, the ledger conversion, and round-trip extraction — with `Inclusion`/`TxInclusion`/`setTxInclusion` exported and a round-trip property test. `cardano-cli` gets a record-pattern fix for the extended `TxBodyContent`. In `cardano-node`'s tx-generator, `genTxWithInclusion` keeps `genTx` backward-compatible while letting callers set the inclusion strategy, and a new `dijkstra-lane-feeder` executable (~400 lines) submits Dijkstra `Urgent`/`Optimistic` transactions over node-to-client with configurable metadata size and independent per-lane chains — built because the stock `tx-centrifuge` only produces Conway transactions.

## Private-network testing (proto-devnet)

With that tooling in place, the prototype can be exercised end-to-end on a private network, well ahead of the formal [testing #69](https://github.com/input-output-hk/tiered-pricing/issues/69) and [node-integration #73](https://github.com/input-output-hk/tiered-pricing/issues/73) milestones. The harness builds on `ouroboros-leios`' `proto-devnet`: a three-node Cardano devnet (process-compose) hard-forked into Dijkstra, with the `dijkstra-lane-feeder` driving load.

As a first proof, a batch of Dijkstra transactions (Optimistic and Urgent) was submitted to a node, forged into a block, and cleared from the mempool — confirming Dijkstra submission and inclusion on the private network.

This is still very early stage: it works in a happy path — transactions flow through both lanes and get forged as expected — without yet stressing congestion or failure cases. The current focus is consolidating that baseline.

Started the CIP draft for transaction urgency signalling: [Transaction Urgency Signalling On Linear-Leios](https://github.com/input-output-hk/tiered-pricing/blob/cip-urgency-signalling/docs/phase-2/CIP-urgency-signalling/README.md).

The draft currently sets out:

- The urgent/standard transaction lanes
- The rule that only urgent transactions can enter Praos / Ranking Blocks, while both urgent and standard transactions can enter Endorser Blocks
- The motivation from increased linear-Leios inclusion latency and latency variance
- The ledger-enforcement requirement for Ranking Blocks
- The mempool implication that urgent transactions need to be valid both at the end of the whole mempool and at the end of the urgent queue
- The dependency on a fee refund mechanism, so users can safely post a buffer against dynamic fee movement
- Initial versioning notes: likely major protocol version / new ledger era, enabled by a hard fork
