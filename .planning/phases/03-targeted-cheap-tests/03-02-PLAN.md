<objective>
Wave 2 of Phase 3: execute the five multi-seed / sensitivity / steady-state /
companion tests that resolve the LIVE entries in the realism-risks register
into BACKED, WEAK, MITIGATED, or LIVE → DISCLOSED verdicts. Five sub-tasks,
all parallel within the wave, all gated on Wave 1's paired_bootstrap.rs
library + chosen N + topology-realistic-150.yaml.

Each sub-task produces one suite YAML, runs it under experiment-suite,
applies paired_bca_ci where applicable, writes a results.md + per-cell
JSON artefacts to `.planning/realism-tests/<test>/`, and **updates the
relevant CLM-NN rows in `docs/phase-2/coverage-check.md` in the same
commit** per CONTEXT.md D-27 (incremental update; no batched end-of-wave
rewrite).

Output: 5 new suite YAMLs (not goldens-pinned per D-25), 4 results.md
files (multi-seed-variance covers TEST-03 + TEST-04 in one directory),
per-cell JSON artefacts, and an incrementally-updated coverage-check.md.

Cross-cutting truths re-stated here so Wave 3 (Plan 03-03) can rely on them:
- `bootstrap_seed` recorded in every <cell>.json (per D-23).
- Per-seed scalar = `priority_retained_value_total + standard_retained_value_total` from RunSummary (per D-24); subsidiary metrics reported as median+IQR but do NOT gate verdict.
- Abbreviations expand on first use in results.md and suite YAML headers.

Parallelism advisory: per CONTEXT.md §"Integration Points", on the dev
machine prefer running each Wave 2 suite sequentially under
`experiment-suite run -P 8` rather than concurrently via
`scripts/run-parallel-suites.sh`; the cross-suite K × intra-suite P
product otherwise exceeds available cores. Raise to cross-suite parallel
only on hardware with N ≥ 16 cores AND > 64 GB RSS.

Wave 2 callers will:
  1. Read run_summary.json per (job, seed) from sim-rs/output/<suite>/<job>/<seed>/
     and extract `priority_retained_value_total + standard_retained_value_total`
     as the per-seed scalar.
  2. For paired comparisons: build samples_a (treatment cell) and samples_b
     (control cell — typically the single-lane EIP-1559 baseline) over the
     same N seeds.
  3. Call paired_bca_ci(...) and paired_delta_summary(...).
  4. Serialize CiResult + DeltaSummary to .planning/realism-tests/<test>/<cell>.json
     via serde_json::to_writer_pretty.

How the Wave-2 caller is structured: There is NO new sim-cli subcommand;
the per-cell aggregation is a small one-shot Rust binary OR a Python
post-processing script that reads JSON files from sim-rs/output/ and
writes JSON files to .planning/realism-tests/. The Wave 2 author may
implement this either as:
  (a) A new dev-binary under sim-rs/sim-cli/src/bin/wave2-aggregate/ (only
      if Rust + paired_bca_ci is desired); OR
  (b) A Python script under sim-rs/scripts/wave2-aggregate.py that calls
      the bootstrap math in Python (statrs equivalent is `scipy.stats`).
The Rust option is preferred because it consumes the exact paired_bca_ci
implementation; the Python option requires the executor to verify
numerical agreement with Rust's paired_bca_ci on a synthetic dataset
before using.

The simplest concrete choice (planner's recommendation): write a tiny
new bin under sim-rs/sim-cli/src/bin/wave2-aggregate/main.rs that reads
a manifest of (cell-name, samples_a-path, samples_b-path, output-path,
alpha, bootstrap_seed) tuples from a YAML file in
.planning/realism-tests/<test>/aggregation.yaml, computes CiResult +
DeltaSummary per cell, and writes the JSON. This keeps the BCa math
in one place and avoids a Python-vs-Rust numerical-equivalence step.
